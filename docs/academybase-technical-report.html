What is AcademyBase?

AcademyBase serves as the operating system for football academies. It replaces fragmented tools — spreadsheets, WhatsApp group notes, manual ledgers — with a unified, role-aware platform accessible from any device.

The platform is fully localized in Arabic with an RTL layout, targeting academy owners and managers in Gulf and Arab markets. It handles the complete operational lifecycle: onboarding new players, tracking attendance session-by-session, managing subscriptions across billing cycles, recording all financial flows, and surfacing actionable insights through dashboards and AI-driven analysis.

Core Value Proposition

Operational Clarity

Real-time view of all active players, their subscription status, upcoming renewals, and attendance streaks — eliminating guesswork from daily academy management.

Financial Transparency

Auto-synchronized revenue from payments, configurable expense categories, staff salary tracking, and branch-level P&L — all in one ledger with override controls.

Multi-Role Team Access

Owner, partner, branch manager, and admin staff roles with distinct data access controls enforced at both UI and database levels via PostgreSQL RLS policies.

AI-Ready Insights

A pure-computation insight engine surfaces critical signals: renewal rate drops, revenue anomalies, expiring subscriptions, and attendance red flags — prioritized by severity.

Current Capabilities

  • Player CRUD with subscription mode support (session-based and monthly billing)
  • Multi-branch management with configurable schedules, pricing, and session calendars
  • Attendance tracking with session-period awareness and optimistic UI updates
  • Financial ledger with auto-sync from payments, staff salaries, and field costs
  • Role-based team collaboration with invitation-based onboarding
  • Insights engine with 5 signal types and severity classification
  • Analytics with trend charts, renewal rates, age distribution, and seasonality
  • Monthly training calendar with match, cancellation, and special event tracking
  • Responsive mobile-first RTL design with dark theme

System Architecture

AcademyBase uses a modern serverless-first architecture with Next.js App Router and Supabase as the sole backend. There is no custom API server — all data access goes directly through Supabase's PostgREST API, secured by Row-Level Security policies.

Frontend Architecture

The frontend is a React 19 single-page application rendered within Next.js App Router. Protected pages are client components ("use client") that fetch data on mount via useEffect + Supabase browser client. Public auth pages (login, register, invite accept) are server components leveraging Next.js SSR.

The layout hierarchy is: Root Layout → App Layout (sidebar + mobile header state) → Page. The App Layout is a client component that manages the sidebar drawer state for responsive behavior.

Backend Architecture

There is no custom API server. Supabase's auto-generated PostgREST API serves all data requests. Authentication uses Supabase Auth (JWT-based). Role and access control is enforced at the database layer via PostgreSQL Row-Level Security — the application layer simply reflects the data Supabase returns.

Server Actions (Next.js) are used for critical write operations that require the service-role key: accepting invitations (acceptInvitationAction), creating academies (setupAcademyAction), and invited user setup (setupInvitedUserAction).

Data Flow Pattern

1

Auth Check (Middleware)

Every request through middleware.ts verifies a valid Supabase session. Unauthenticated users are redirected to /login. Server Actions bypass this redirect to prevent client-side errors.

2

Academy ID Resolution (Cached)

resolveAcademyId() reads profiles.academy_id for the current user. The result is cached as a module-level Promise — all concurrent callers share a single in-flight query. Only 1 database call per session.

3

Parallel Data Fetch

Pages execute Promise.all([...libFunctions]) where each lib function uses the cached academy_id. This achieves parallel DB queries with a single resolved identifier.

4

RLS Enforcement (Database)

Supabase applies RLS policies on every query. SECURITY DEFINER helper functions resolve user context without recursive policy evaluation. Non-authorized users receive empty result sets, not errors.

5

State + Render

Data lands in React state. useMemo computes derived metrics (renewal rate, KPIs). Charts render via Recharts (tree-shaken). Optimistic UI updates for attendance toggling.

Architectural Philosophy

Zero Backend Servers

No Express, no Django, no custom API. All data operations go through Supabase PostgREST. This eliminates an entire infrastructure tier and reduces operational overhead to near zero.

Zero State Management Library

No Redux, no Zustand, no Jotai. Plain React hooks (useState, useMemo) with Supabase as the single source of truth. Eliminates boilerplate and synchronization bugs.

Zero localStorage

All user data persists in Supabase only. Session-scoped data uses module-level memory (cleared on logout). This prevents stale data bugs across login sessions and is more secure.

Security at the Database

Row-Level Security enforces authorization at the data layer. The frontend doesn't need to implement access control — RLS returns only the rows each user is permitted to see, regardless of how queries are made.

✓

This stack combination enables a single developer to build and operate a production SaaS. Supabase handles authentication, database hosting, backups, and the REST API. Next.js + Vercel handle global CDN, SSR, and serverless compute. The total infrastructure cost at moderate scale is under $50/month.

Row-Level Security Design

The most technically complex part of the database is the non-recursive RLS design. A naive approach of referencing academy_members inside policies creates a circular dependency (error 42P17: infinite recursion detected). AcademyBase solves this with a canonical "source of truth" pattern and SECURITY DEFINER helper functions.

⚠

The Recursion Problem: If branches.SELECT policy queries academy_members, and academy_members.SELECT policy queries academies, and academies.SELECT queries academy_members again — PostgreSQL enters infinite recursion. This is a common Supabase pitfall.

Solution: SECURITY DEFINER Helpers

SQL
-- Helper runs as postgres (BYPASSRLS) — breaks recursion
CREATE OR REPLACE FUNCTION public.my_academy_id()
RETURNS uuid
LANGUAGE sql STABLE SECURITY DEFINER
AS $$
  SELECT academy_id FROM profiles
  WHERE user_id = auth.uid();
$$;

-- RLS policy on branches uses helper, NOT inline subquery
CREATE POLICY "branches_select" ON branches
  FOR SELECT USING (academy_id = my_academy_id());

Migration Strategy

33 sequential migration files form the complete schema history. Each file is idempotent where possible (DROP IF EXISTS + CREATE OR REPLACE). Migration 14 is the critical fix migration that resolves the RLS recursion bug and can be safely re-run on any environment.

Key Indexes

  • UNIQUE(academy_id, user_id) on academy_members — prevents duplicate membership
  • UNIQUE(academy_id, player_id, date) on attendance — upsert conflict target
  • UNIQUE(academy_id, auto_key) on finance_tx — idempotent auto-sync entries
  • UNIQUE(branch_id, date) on sessions — per-day field cost record
  • INDEX on payments(player_id, date) — renewal rate queries scan by player

06 — Finance

Financial System

The financial engine is the most sophisticated module. It combines auto-generated entries from operational data with manual entries, giving academy owners a real-time view of their financial position.

Transaction Categories

Revenue (إيراد)

  • Subscription payments (auto-synced from payments table)
  • Aggregated by month + branch
  • Separated by new vs. renewal payments
  • Manual revenue entries (sponsorships, events)

Expenses (مصروف)

  • Staff monthly salaries (auto-synced from staff records)
  • Field rental costs (per-session from sessions table)
  • Manual expense entries (equipment, utilities)
  • Staff attendance adjustments (bonuses/deductions)

Auto-Sync Engine

When a user opens the finance page for a given month, the system automatically generates transactions from operational data via upsertAutoFinanceTx(). This uses INSERT ... ON CONFLICT (academy_id, auto_key) DO UPDATE — making every sync idempotent. Existing manual entries are never overwritten.

TypeScript
// Auto-sync generates entries like:
{
  auto_key: "payment_2026-03_branch_uuid",  // unique per month+branch
  type: "إيراد",
  amount: 1250,                        // sum of all branch payments
  source: "auto",
  month: "2026-03"
}

{
  auto_key: "salary_2026-03_staff_uuid",   // unique per month+staff
  type: "مصروف",
  amount: 800,                         // monthly_salary from staff record
  source: "auto",
  month: "2026-03"
}

Suppression System

Instead of deleting auto-generated entries (which would cause them to regenerate on next sync), users can "suppress" entries. This sets source = "suppressed" and amount = 0. On subsequent syncs, the conflict resolution skips suppressed entries, preserving the user's intent.

Session Deductions

The sessions table tracks field rental cost per training session. When a calendar event is marked with deduct_sessions = true, it contributes to the branch's cost calculation for that month. This enables per-session financial tracking alongside monthly aggregates.

Salary Calculation

Staff salaries are fixed monthly amounts stored in staff.monthly_salary (non-negative constraint enforced by DB). Staff attendance adjustments (bonuses, deductions tracked in staff_attendance table) can modify the effective monthly amount. The auto-sync reads both and computes net salary per staff member.

Renewal Rate Algorithm (Fixed)

The renewal rate calculation required a significant bug fix. The original implementation counted payment rows by kind field, which could produce a count of 2 for a single player who made two payments in a month. The corrected algorithm:

  • Groups payments by player_id using a Map (O(n) instead of O(n×m))
  • Finds each player's first-ever payment date (earliest date across all payment history)
  • Classifies a player as "new" if their first payment falls in the selected month
  • Classifies a player as "renew" if they have payment history before the selected month
  • Counts distinct players, not payment rows

08 — Security

Authentication & Security

Security is enforced at three layers: middleware (route protection), RLS (data access), and service-role server actions (privileged operations). The design follows a zero-trust data model.

Authentication Flow

1

Sign Up / Sign In

Standard email/password via Supabase Auth. Returns a JWT token stored in the browser (cookies via Supabase SSR package). Email confirmation supported.

2

Middleware Route Guard

middleware.ts runs on every request. Reads the Supabase session from cookies. If no valid session, redirects to /login?next={path}. Server Actions (POST with next-action header) bypass the redirect to prevent client errors.

3

Academy Registration

Server Action setupAcademyAction() uses service-role key to create: academy record → profile (with academy_id) → academy_members(role="owner"). Atomic within a single Server Action scope.

4

JWT Token Refresh (iOS Fix)

iOS Safari's navigator.locks API has a 10s timeout. To prevent contention when multiple components race to call getSession(), a singleton Supabase client is used in the browser. Only one lock holder at a time — no timeout errors on iPad/iPhone.

Invitation System

Team members are onboarded via a 32-byte random hex token (256-bit entropy — brute-force infeasible). The invite link is /invite/{token}. On acceptance:

  • Server validates email match between invite and current user
  • Upserts academy_members with the specified role
  • Updates profiles.academy_id (the canonical data store)
  • Invite token expires after the configured expires_at timestamp

Security Posture Assessment

Strengths

  • RLS enforced at DB layer — bypass is impossible without service-role key
  • Service-role key used only in Server Actions (never exposed to client)
  • 256-bit invite tokens (cryptographically secure)
  • No sensitive data in localStorage or client state
  • Singleton client prevents iOS Safari lock exploitation

Areas for Improvement

  • No rate limiting on auth endpoints (Supabase defaults apply)
  • No audit log for admin actions (player delete, finance override)
  • Branch-scoped staff can theoretically edit players in other branches via direct PostgREST calls
  • No 2FA support currently

Scaling Concerns

⚠

Full table loads: Pages like Players and Finance currently fetch all rows for the academy and filter in the browser. With 1000+ players or 5 years of finance history, this becomes a scaling bottleneck. Server-side pagination is not yet implemented.

⚠

Attendance table growth: At 100 players × 3 sessions/week × 52 weeks = 15,600 rows/year. A 5-year academy with 500 players would have ~780,000 attendance rows. The attendance page loads all rows for a month, which is manageable, but the attendance trend query in Statistics loads potentially months of data.

ℹ

Insights not memoized at page level: computeInsights() re-runs on every render cycle where data exists. While it's a pure function and relatively fast, wrapping it in useMemo would prevent redundant computation on unrelated state updates.

11 — Weaknesses

Product Weaknesses

An honest assessment of current limitations across UX, architecture, data design, and performance — to guide the next improvement cycle.

UX Weaknesses

No Global Search

There is no cross-module search. Finding a specific player requires navigating to the Players page and using the filter. A cmd+K style command palette would significantly improve navigation speed for power users managing large academies.

No Notification System

The insights engine identifies critical signals (expiring players, attendance drops) but there is no proactive notification — not even an in-app notification badge. Users must actively visit the insights page to see alerts.

Arabic-Only Interface

All UI strings are hardcoded in Arabic. There is no i18n layer. Expanding to non-Arabic markets or international coaches on the team would require significant string extraction work.

No Bulk Operations

Players can only be acted upon one at a time. For academies that want to send renewals to 50 expiring players simultaneously, or mark a class of players as attended after a session, there is no multi-select or bulk action capability.

Architecture Weaknesses

No Server-Side Pagination

All data is fetched client-side with RLS filtering. As academies grow (500+ players, 3+ years of finance history), initial page loads will degrade. No pagination, virtual scrolling, or server-side filtering is implemented.

No Optimistic Updates Outside Attendance

Only the attendance toggle has optimistic updates. Player edits, payment creation, and finance entries all show loading states with no immediate UI feedback. This creates a noticeable lag feel even when the DB operation succeeds quickly.

Concurrent Finance Sync Risk

If two users (owner + admin) open the finance page simultaneously, both trigger upsertAutoFinanceTx() for the same month. The UNIQUE conflict resolution handles this correctly, but under high concurrency, there's a theoretical risk of duplicate entries if the conflict key logic changes.

No Real-Time Updates

Supabase provides WebSocket-based Realtime subscriptions, but none are used. If two team members are viewing the players list simultaneously and one adds a player, the other won't see the update until they refresh.

Data Design Weaknesses

Subscription Period Gaps

When a player renews, the new start_date might create a gap between the old subscription end and the new start. The attendance page handles this correctly (checks all periods), but the Players page status display uses player.end_date which is overwritten on each renewal, losing the previous period's end date from the player record itself.

Branch_ids Array on Staff

Staff assignment to branches uses a PostgreSQL UUID array (branch_ids uuid[]). This is not normalized — it's not enforced by foreign keys, there's no cascading behavior on branch delete, and it can't be efficiently joined with SQL. A junction table would be more robust.

Performance Weaknesses

Finance Page is the Heaviest Page

At ~1,500 lines, the finance page loads the most data and performs the most complex client-side aggregation. Auto-sync logic that runs on mount makes 5–7 separate upsert calls. As data grows, this page will be the first to show noticeable slowdowns.

Insights Not Wrapped in useMemo

computeInsights() iterates all payments, players, and attendance records on every call. Without useMemo, this recomputes on every state update in the page — including unrelated state like modal open/close events.

12 — Improvements

Improvement Opportunities

Prioritized, actionable improvements organized by impact area. Each recommendation includes implementation guidance and expected benefit.

High Priority — Architecture

1

Server-Side Pagination for Players and Finance

Implement range(offset, limit) Supabase queries with server-side filtering. Use Next.js URL search params for page state. Expected: 80% reduction in initial data load for large academies. Required for scaling beyond 500 players.

2

Database Indexes on High-Traffic Queries

Add composite indexes: (academy_id, date) on attendance and payments; (academy_id, end_date) on players; (academy_id, month) on finance_tx. Expected: 5–10× query speed improvement at scale.

3

Supabase Realtime Subscriptions

Subscribe to players and attendance table changes via supabase.channel(). Enable multi-user collaboration without refresh. Particularly impactful for attendance taking where multiple staff might mark the same session.

Medium Priority — UX

4

Global Command Palette (Cmd+K)

Implement a fuzzy-search command palette over players, branches, and navigation. Libraries: cmdk or custom implementation. Transforms player lookup from navigate→filter→find to instant search. Critical for large academies.

5

In-App Notification System

Add a notification bell in the header that runs a lightweight version of computeInsights() on mount. Show count badge for critical signals. Toast notifications for actions. Enables passive awareness without visiting insights page.

6

Bulk Player Actions

Add multi-select checkboxes on the players list. Enable bulk: renew, extend, message, export to CSV. Especially valuable at end-of-month for bulk renewal processing. Reduces 50 individual modal interactions to 1 bulk action.

7

Optimistic Updates Across All Forms

Extend the optimistic update pattern from attendance to player CRUD, payments, and branch edits. Update local state immediately on form submit, revert on error. Expected: perceived performance improvement of 200–800ms on all write operations.

Lower Priority — Code Quality

8

Wrap computeInsights() in useMemo

Add useMemo(() => computeInsights(input), [players, payments, attendance, branches]) in both the dashboard and insights pages. Prevents recomputation on unrelated state updates. Small change, immediate gain.

9

Normalize Staff Branch Assignment

Replace staff.branch_ids uuid[] with a junction table staff_branch_assignments(staff_id, branch_id). Enables FK constraints, cascade on branch delete, and efficient branch-based staff queries.

10

Extract Common Page Shell Component

All pages share a pattern: loading state → error state → data display. Extract a <PageShell loading={} error={}> component to reduce boilerplate across the 8 main pages. Reduces ~50 lines of repetitive code per page.

13 — Roadmap

Future Roadmap

Strategic feature directions that would transform AcademyBase from an operational tool into an intelligent academy intelligence platform.

Phase 1 — Enhanced Intelligence (Q2 2026)

Advanced AI Insights

Extend the insights engine with LLM-generated natural language explanations. Use Claude API to produce actionable recommendations in Arabic: "3 players from Branch A have missed 4+ consecutive sessions. Consider sending a WhatsApp follow-up this week."

  • Connect computeInsights() output to Claude API
  • Arabic-language response generation
  • One-click action triggers from insight cards
  • Weekly digest email with top 5 insights

Predictive Renewal Analytics

Use historical payment data to predict which players are likely to churn. Train a simple logistic regression model on: attendance rate, payment timeliness, session completion rate, and absence streaks.

  • Churn probability score per player (0–100%)
  • Risk segmentation: high/medium/low
  • Renewal probability chart on dashboard
  • Proactive re-engagement list generation

Phase 2 — Platform Expansion (Q3 2026)

Academy Benchmarking

Anonymized cross-academy benchmarking showing how an academy's renewal rate, revenue per player, and attendance rate compare to similar academies by size and geography. Requires opt-in data sharing and aggregate-only API.

  • Percentile ranking (top 25% in region)
  • Benchmark dashboard card
  • Gap analysis vs. top performers
  • Best practice recommendations

Financial Forecasting

12-month revenue and expense forecast based on historical trends, current subscription mix, and seasonal patterns identified in the Statistics module.

  • Monte Carlo simulation with confidence intervals
  • Scenario modeling (10% churn vs. 20% growth)
  • Break-even analysis per branch
  • Staff salary vs. revenue ratio alerts

Phase 3 — Ecosystem (Q4 2026)

Parent & Player Portal

A separate read-only portal where parents can view their child's attendance record, payment history, and upcoming schedule. Reduces inbound WhatsApp messages to academy staff by 60–70% based on industry benchmarks.

WhatsApp Integration

Automated WhatsApp messages via WhatsApp Business API: renewal reminders (7 days before expiry), absence alerts (3 consecutive misses), payment confirmations. All triggered from the insights engine signals.

Mobile Native App

React Native or Expo wrapper for the attendance and calendar modules. Enables offline-first attendance marking that syncs when connectivity is restored. Critical for field operations with poor internet.

Player Performance Tracking

Add a performance assessment module allowing coaches to score players on configurable attributes (speed, technique, teamwork). Progress tracking over time. Feeds into AI insights about player development trajectories.
